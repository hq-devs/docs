# T13 — 功能点与数据契约（原型对照草案）

> 本文以 `ai-point-web` **原型实际功能**为准，与目标后端架构（云端无状态计算 · 数据全本地 · per-session xboard 容器）**相互对照**，推导一条主线：**功能点 → 需要的数据 → 后端提供 → API 草案 → 数据归属**，用于把数据模型逐步定下来。
> **定位原则**：原型是**功能点的参考来源，不是复刻规范**。本文关注「有哪些功能、每个功能消费/产出什么数据」，**不追求 1:1 还原原型的字段名 / 结构 / 交互细节**；后端 schema 可按更干净的设计归并、改名、简化或偏离原型。
> **状态**：草案，**非最终 schema**。最终本机文件 schema 归 [T3](T3-数据库与数据模型设计.md)，Bridge 方法与事件 payload 归 [T4](T4-API与桌面Bridge设计.md)，架构与存储边界见 [T1](T1-系统架构设计.md) / [T12](T12-数据与物料存储边界设计.md)。
> 来源：原型 `ai-point-web/prototype/`（index / overview / workbench + `assets/mock-data.js`）。

## §0 数据归属图例

目标态只有两个**持久域**：① 本机权威（持久）；② 云端会话临时态（会话结束即销毁）。外加流式事件与本机配置。

| 标签 | 含义 |
|---|---|
| **本机权威** | 工程师本机工程目录持有的真身（点表 DSL、设备配置、调试结果、决策）|
| **云端会话临时** | 后端在生成 / 调试会话存活期间持有的中间产物（jobfs scratch + 内存会话注册表 + xboard 容器），结束即清 |
| **流式事件** | 通过 gRPC Streaming → Bridge `EventsEmit` 推给前端的实时数据（进度 / 实时值 / 帧 / metrics）|
| **本机配置** | 客户端本地配置（版本、规则包版本、工程目录、平台连接）|
| **云端只读分发** | 规则包等只读分发物，非业务状态 |

## §1 功能点 → 数据 → 后端 → API（三条主线）

> API 列为**草案**（gRPC 服务 / 本机 Bridge 方法），命名以表意为先，最终以 T4 为准。

### 主线 A · 工程与入口（P1 启动 / P2 总览）

| 功能点 | 需要的数据 | 后端提供 / API 草案 | 归属 |
|---|---|---|---|
| 客户端启动 / 配置 | `version`、`rulePack`、`defaultProjectDir`、`onboarded`、`projectDir` | 下发客户端版本与规则包版本；配置本机持有<br/>`MetaService.GetClientMeta` · `RulePackService.GetRulePackVersion` | 本机配置 |
| 平台连接（在线点表服务器）| `server`、`projectKey`、`account`、`connected`、`lastSync` | 测试连接 / 保存目标（`remote` 与 `submit` **合一**）<br/>`PlatformService.TestConnection` · `SaveTarget` | 本机配置 |
| 最近工程 / 工程列表 | `id`、`name`、`client`、`path`、任务状态分布 `dist`、`lastOpened` | Bridge 枚举本机 `ptw-project.json`<br/>`App.ListProjects`（本机）| 本机权威 |
| 新建 / 打开工程 | `name`、`client`、`path` | Bridge 本地创建 / 读取 `ptw-project.json`<br/>`App.CreateProject` · `OpenProject` | 本机权威 |
| 设备任务列表（总览）| `id`、`name`、`vendor`、`model`、`protocol`、`board`、`status`、`read/write/err`、`updated` | Bridge 枚举本机 `tasks/*/task.json`<br/>`App.ListTasks`（本机）| 本机权威 |

### 主线 B · 生成闭环（P3 import → parsing → confirm → ready）

| 功能点 | 需要的数据 | 后端提供 / API 草案 | 归属 |
|---|---|---|---|
| 导入协议文档 · 建点表 | 上传文件；`accept[]`、`sampleFile` | 上传 → OCR(MinerU) → 生成 Job（无状态请求）<br/>`DocumentService.UploadDocument(stream)` → `GenerationService.Generate` | 云端会话临时 |
| 生成进度 | `stages[]{name,state,tip}`、`summary{read/write/suspect/fail}` | Job 阶段进度推送<br/>`GenerationService.StreamProgress` | 流式事件 |
| 协议文档管理 | `id`、`name`、`role`、`pages`、`size`、`state`、`uploaded` | 文档登记 + 解析态；原件本机留存<br/>`DocumentService.List` · Bridge 本机 `protocol/` | 本机权威 |
| 原文 / OCR 对照预览 | `docId`、`no`、`title`、`raw[]`、`ocr[]` | OCR 结果（证据定位源），下发本机副本<br/>`EvidenceService.GetDocPages` | 云端会话临时 |
| 读 / 写测点表 | 协议字段 + 调试字段 + DTDL + `suggestion` | 生成产物渲染为 JSON DSL，下发本机权威<br/>`PointTableService.GetPointTable` → 本机 `point_table.json` | 本机权威 |
| 命令表 / 设备信息表 | `cmd/start/end`；`name/value/desc` | 随 DSL 一并下发本机<br/>`PointTableService.GetPointTable` | 本机权威 |
| 澄清遮罩门 | `q`、`evidenceRef(docId+page)`、`impact`、`opts[]`、`recommend`、`reason`、`points[]`、`resolved` | 生成澄清项；回收答案后 merge 重算<br/>`ClarificationService.List` · `AnswerClarification`(MergeWithAnswer) | 云端会话临时 |
| 变更影响分析（增量）| `docId`、`items[]{type,state,title,detail,evidence,apply(add/scale/na)}` | 新旧文档增量分析；接受 / 拒绝落点表<br/>`IncrementalService.Analyze` · `DecideChange` | 云端会话临时 |
| AI 修正建议（点 / 命令）| `title`、`reason`、`diff[]{field,from,to}`、`fix{...}`、`evidenceRef`、`fieldIssue` | 调试 / 校验产出，嵌入点表与命令<br/>随 `DebugService` 结果 | 云端会话临时 |

### 主线 C · 调试闭环（P3 右栏）

| 功能点 | 需要的数据 | 后端提供 / API 草案 | 归属 |
|---|---|---|---|
| 启动调试 · 起会话容器 | 上传 `point_table.json` + `device.json`（`comm`·`addr`·`transfer`）| 渲染会话 xlsx → 从 warm pool 绑定 per-session xboard 容器 → 注入配置 + 建隧道<br/>`DebugService.StartDebug` | 云端会话临时 |
| 实时读取 / 轮询 | `ReadPoint.raw`·`rawVal`·`val`·`state`（逐点回填）| 采集回包逐点推送，写本机调试态<br/>`DebugService.StreamRealtime` | 流式事件 |
| 收发报文 | `dir(TX/RX)`·`t`·`hex`·`note`·`cmd`·`err` | 帧流推送 + 本机 `hex_log.json` 落地<br/>`DebugService.StreamFrames` → 本机 `debug/{id}/hex_log.json` | 流式事件 |
| 命令画像 | `respTypical/Worst`·`jitter`·`recInterval/Timeout/Retry`·`samples`·`successRate`·`frameReq/Resp` | 采集统计聚合回写<br/>`DebugService.GetCommandProfile` | 流式事件 |
| 手动发包 | req hex 帧 → 透传 → 解析 RX | Modbus 透传（经隧道回工程师 Bridge → 真实设备）<br/>`DebugService.SendFrame` | 流式事件 |
| 通信参数 | `mode`·`port`·`baud`·`dataBits`·`parity`·`stopBits`·`ip`·`tcpPort`·`timeout`·`retry`·`interval` | 本机权威；调试时注入会话容器采集配置<br/>本机 `device.json` + `DebugService.ApplyProfile` | 本机权威 |
| 调试链路状态 | `device/instance/tunnel` 三灯 · `latency` · `tx/rx` · `instanceId` · `ttl` | 隧道 / 容器健康与计量推送<br/>`DebugService.StreamLinkStatus` | 流式事件 |
| 自收敛 loop 自动应用 → 重部署 → 回采（每轮，外壳执行）| 本轮假设 `FieldDiff` + 收敛集 | 经安全门自动改 DSL → 重渲染 xlsx → `/updateTemplate` 驱逐缓存重测 → 回采验证不降级（无人工 apply 按钮，T2 §5）| 本机权威 |
| 记录现场问题 | `FieldIssue` / 调试报告（原型未建模）| 落本机调试报告<br/>Bridge 本机写 `debug/{id}/report.json` | 本机权威 |
| 收敛落定 → 新版本 + 逐轮审计 | `converged/partial` · `converged_seqs/unconverged_seqs` · `dsl_version` · 逐轮 `AppliedChange` | loop 收敛后落定新版本下发本机权威；逐轮自动应用只读审计<br/>`DebugService.GetSession` · `GetRounds` → `GetPointTable(vN)` | 本机权威 |
| 质量门禁 / 确认提交 | `gatePass`（读点无 suspect/fail/unread 且无未决澄清）· 提交目标 | 门禁校验 + 发布平台<br/>`WorkflowService.Submit` · `Publish` | 云端会话临时 |

> **质量门禁 `gatePass`（提交前后端必须校验）**：读点 `state` 无 `suspect`/`fail`/`unread`，且无未 resolved 澄清项。

| `FrameEvent`（HexLog）| `dir` · `t` · `hex` · `note` · `cmd` · `err` | 流式事件 |

| `LinkStatus` | 三灯 · `latency` · `tx/rx` · `instanceId` · `ttl` | 流式事件 |

## §4 关键决策与待定缺口

1. **`device.json` 要按原型功能点补全（比现有 T3 丰富）**
   原型 `device` 含 `comm`（串口/TCP 全套：`port/baud/dataBits/parity/stopBits/ip/tcpPort/timeout/retry/interval`）、`addr`、`link`、`fingerprint`。当前 T3 的 `DeviceConfig` 仅覆盖采集必需项，需以**功能点**为准扩充本机 `device.json`（字段名可规范化，不必照搬原型）。

2. **最大缺口：五类流式事件 payload 未定义**
   生成进度 · 逐点实时回填（`raw/rawVal/val/state`）· 报文帧 · 链路 metrics · 命令 profile —— 原型只有「展示态字段」，没有「传输态契约」。这是闭环最关键、最该先定的（归 T4/T8）。

3. **`remote` 与 `submit` 两套平台连接字段合一**
   P1/P2 用 `remote.*`，P3 工作台提交用 `submit.*`，语义重叠。契约里合并为单一「平台连接 / 提交目标」实体。

4. **`evidence` 与 `docPages` 必须一起定**
   澄清 / 变更 / suggestion 全部带 `evidenceRef(docId+page)`，要能定位回 `docPages` 的 OCR / 原文；证据链 schema 与文档页 schema 是一套。
